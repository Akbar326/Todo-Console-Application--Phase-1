# Data Model: Todo Console Application

## Phase 1: Data Structure Design

### Task Entity

```python
from dataclasses import dataclass
from typing import Optional

@dataclass
class Task:
    id: int
    title: str
    description: str
    completed: bool = False
```

**Attributes:**
- `id`: Unique identifier for the task (integer)
  - Auto-generated by TaskManager
  - Sequential, starts from 1
  - Used for lookup and operations
- `title`: Required task title (string)
  - Non-empty string validation required
  - Displayed prominently in task lists
- `description`: Optional task details (string)
  - Can be empty string if not provided
  - Provides additional context for the task
- `completed`: Boolean status indicator
  - Default value: False
  - True when task is marked complete
  - False when task is marked incomplete

### TaskManager Internal Storage

```python
class TaskManager:
    def __init__(self):
        self._tasks: Dict[int, Task] = {}
        self._next_id: int = 1
```

**Storage Strategy:**
- Dictionary with task ID as key and Task object as value
- `_next_id` tracks the next available ID for auto-increment
- Provides O(1) lookup time for task operations
- Maintains data integrity through encapsulation

### Data Flow Patterns

**Add Task:**
1. User provides title and description
2. TaskManager creates new Task with next available ID
3. Task is stored in `_tasks` dictionary
4. `_next_id` is incremented
5. Task object is returned

**Update Task:**
1. User provides task ID and new values
2. TaskManager looks up existing task by ID
3. Task attributes are updated
4. Updated task is stored back in dictionary

**Delete Task:**
1. User provides task ID
2. TaskManager removes task from dictionary
3. Returns success/failure status

**Get Tasks:**
1. TaskManager returns list of all Task objects
2. Can be filtered by completion status if needed
